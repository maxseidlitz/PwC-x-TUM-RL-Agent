## Architecture

The project trains a PPO agent to optimize inventory replenishment decisions for a single-echelon supply chain. There is no test suite; validation is done by running the app and inspecting KPIs.

### Core module: `inventory_ppo.py`

All domain logic lives in this single file:

- **`SingleEchelonEnv`** — Gymnasium environment. Observation: `[current_inventory, pipeline[0..lead_time-1], forecast[0..n_forecast_weeks-1]]`, all normalized by `max_order_qty`. Action: continuous `[0,1]` scaled to order quantity. Reward: negative total cost / 1000.
- **`run_training_pipeline(config)`** — End-to-end entry point: loads data, trains PPO via stable-baselines3, runs future projection, computes base-stock baselines, saves artifacts to `runs/<timestamp>_<slug>/`.
- **`run_base_stock_policy(S, ...)`** — Simulates a fixed base-stock policy (order up to `S - inventory - pipeline`) for comparison.
- **`compute_base_stock_baselines(config, ...)`** — Produces three variants (conservative, middle, aggressive) with `S = avg_demand × (lead_time + 0/1/2)`.
- **`load_data(file_path, product, location, scenario)`** — Reads the Excel file. The `v3` format has a `Scenario` column in the `Forecast` sheet; older files do not. Week labels are `ww.yyyy` strings.
- **`save_run_artifacts(...)`** — Writes `config.json`, `model.zip`, `results.xlsx`, `results.png`, and `records.json` to the run directory.
- **`visualize_results(...)`** — Generates a 5-panel matplotlib figure (KPI row, inventory+demand, order qty, cost breakdown, cumulative cost). Imports color tokens from `ui/tum_theme.py` with a fallback to hardcoded defaults so it works outside the UI context.

### UI (`ui/`)

- **`app.py`** — Streamlit entry point. Two tabs: *Current Run* (sidebar config → train → Plotly dashboard) and *Compare Runs* (filter + overlay up to 5 saved runs).
- **`dashboard.py`** — Builds interactive Plotly figures (`build_dashboard_figure`, `build_comparison_figure`). Imports color tokens from `tum_theme.py`.
- **`run_loader.py`** — Reads `runs/*/config.json` and `runs/*/records.json` to enumerate and load saved runs. `RunSummary` is a lightweight listing; `LoadedRun` carries full records.
- **`tum_theme.py`** — TUM corporate design color constants and Streamlit CSS injection (`inject_tum_styles`, `render_tum_header`).

### Data flow

```
Excel file
  └── load_data()  →  demand[], forecast[], lead_time, initial_inventory, week_labels
        ├── SingleEchelonEnv  →  PPO.learn()  →  trained model
        │     └── evaluate / run_future_projection  →  records[]
        ├── compute_base_stock_baselines()  →  bs_results[]
        └── save_run_artifacts()  →  runs/<id>/{config.json, model.zip, records.json, ...}
                                        └── run_loader.py  →  Streamlit Compare tab
```

### Run artifacts (`runs/<timestamp>_<slug>/`)

| File | Purpose |
|---|---|
| `config.json` | All hyperparameters + summary KPIs |
| `model.zip` | Stable-Baselines3 PPO model |
| `records.json` | Serialized weekly step records + base-stock results (used by UI) |
| `results.xlsx` | Multi-sheet Excel export |
| `results.png` | Static matplotlib dashboard |

### Key design decisions

- **`max_order_qty` auto-detection**: if 0 or out of range, it is set to `3× peak weekly demand` to keep the action space tractable.
- **`n_steps` auto-capping**: reduced so at least 10 PPO updates occur within the requested `timesteps`.
- **Multi-scenario support**: when multiple scenarios are selected, future projections are run per-scenario and averaged week-by-week (`_average_records`).
- **Scenario-aware forecasts**: `v3` Excel format adds a `Scenario` column to the `Forecast` sheet; `load_data` detects this automatically via `has_scenario_col`.
